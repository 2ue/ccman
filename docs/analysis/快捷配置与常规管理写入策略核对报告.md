# 快捷配置与常规管理写入策略核对报告

## 1. 结论摘要

本次核对采用以下统一标准：

- **快捷配置 / 一键配置 / setup 类入口**：应该走**覆盖写入（overwrite）**
- **常规管理 / provider CRUD / use 切换类入口**：应该走**增量写入（merge）**

基于当前代码，结论如下：

1. **主链路 `ccman cx/cc/gm/oc/openclaw` 已基本符合“常规管理=增量”**
2. **主链路 `ccman gmn/gmn1/gmncode` 已基本符合“快捷配置=覆盖”**
3. 但仍存在几个重要问题：
   - `ccman gmn*` 在“复用已存在 provider”时，会先触发一次 `edit()`，如果该 provider 恰好是当前激活 provider，就会先发生一次**增量写入**，然后再 `switch(..., { mode: 'overwrite' })` 做覆盖写入；最终结果是覆盖，但实现过程不是“纯覆盖”
   - `aicoding` 仍保留“保护模式（默认）/ 全覆盖模式”双模式；如果把它也视为快捷配置类入口，则**不符合**“类似 `ccman gmn` 的入口统一走覆盖”的目标
   - 历史脚本 `scripts/setup-gmn.mjs` 的**代码行为**与**文档口径**冲突：代码是覆盖，文档写成默认保护
   - `add-openclaw-support` 旧提案/spec 仍写着“OpenClaw 切换时直接覆盖写入”，这和现在项目已经落下的“常规管理增量、快捷命令覆盖”策略冲突

因此，若以你当前希望的最终规则来判断：

- **主产品主链路方向是对的**
- **还没有完全收口**
- 当前最需要关注的是：
  - 快捷入口是否存在“先 merge 再 overwrite”的中间副作用
  - 所有“类似 gmn 的快捷入口”是否统一到了覆盖语义
  - 旧文档 / 旧提案 / 历史脚本文档是否仍在输出相反口径

## 2. 本次采用的核对标准

### 2.1 你希望的规则

你的需求可以抽象成一条“**按入口分类，而不是按工具分类**”的规则：

- 决定写入方式的维度，不是 `Codex / Claude / Gemini / OpenCode / OpenClaw`
- 而是：**用户通过什么入口触发写入**

也就是说：

#### A. 快捷配置类入口

应该统一是：

- `ccman gmn`
- `ccman gmn1`
- `ccman gmncode`
- 其他同性质的一键配置 / setup / bootstrap 入口

这类入口的目标是：

- 快速落下一套“已知可用”的配置
- 不强调保留用户原配置
- 因此应统一走 **overwrite**

#### B. 常规管理类入口

应该统一是：

- `ccman cx ...`
- `ccman cc ...`
- `ccman gm ...`
- `ccman oc ...`
- `ccman openclaw ...`
- 对应 Desktop 中的常规 provider 管理
- 对应交互菜单里的常规 provider 管理

这类入口的目标是：

- 日常维护 provider
- 切换当前 provider
- 尽量保留现有配置里的非托管字段
- 因此应统一走 **merge**

## 5. 哪些能力还没有完全实现

这里的“没有完全实现”，不是说完全没有代码，而是说**还没有完全达到你想要的严格规则**。

## 5.1 `ccman gmn*` 不是“纯覆盖路径”

### 当前实现

- `packages/cli/src/commands/gmn.ts:557`

```ts
const provider = existing
  ? manager.edit(existing.id, { name: providerName, baseUrl, apiKey: resolvedApiKey })
  : manager.add({ name: providerName, baseUrl, apiKey: resolvedApiKey })

manager.switch(provider.id, { mode: 'overwrite' })
```

### 问题

当 `existing` 存在时，先执行 `manager.edit(...)`。

而 `edit()` 内部，如果被编辑的是当前激活 provider，会先触发一次默认 writer：

- `packages/core/src/tool-manager.ts:366`

```ts
if (config.currentProviderId === id) {
  toolConfig.writer(provider)
}
```

这意味着：

- 如果 `gmn` 复用的是**当前正在使用的 provider**
- 那么流程会变成：
  1. `edit()` 先做一次 **merge**
  2. `switch(..., { mode: 'overwrite' })` 再做一次 **overwrite**

### 结论

最终落盘结果大概率还是覆盖后的结果，
但从“严格按入口语义执行”的角度看，这仍然是**未完全实现**。

你要的是：

- 快捷入口 = 直接覆盖

而现在是：

- 快捷入口 = **可能先增量，再覆盖**

这是当前最核心、最应该修的一个实现缺口。

## 5.2 `aicoding` 如果也算快捷入口，则尚未统一到覆盖语义

### 当前实现

`aicoding` 默认仍然是保护模式：

- `packages/aicoding/bin/aicoding.js:355`

```js
{ name: '保护模式（默认，尽量保留现有配置）', value: 'protect' },
{ name: '全覆盖模式（覆盖配置，谨慎使用）', value: 'overwrite' },
```

OpenCode 在保护模式下明确走 merge：

- `packages/aicoding/bin/aicoding.js:503`


OpenClaw 是特例，始终覆盖：

- `packages/aicoding/bin/aicoding.js:606`


### 结论

如果你的规则是：

- “凡是类似 `ccman gmn` 的快捷配置入口，都应该覆盖”

那么 `aicoding` **当前不符合**：

- Codex / OpenCode 仍然保留默认保护模式
- 只有 OpenClaw 被强制覆盖

因此：

- **`aicoding` 的快捷语义尚未统一**

## 6. 目前代码里存在的冲突

## 6.1 旧 OpenClaw spec 与现行写入策略冲突

### 旧 spec 怎么写

- `openspec/changes/add-openclaw-support/specs/openclaw-integration/spec.md:17`

旧 spec 明确说：

- OpenClaw 切换 provider 时，应直接覆盖写入目标文件

### 当前代码怎么做

OpenClaw writer 已支持两种模式：

- overwrite：`packages/core/src/writers/openclaw.ts:263`
- merge：`packages/core/src/writers/openclaw.ts:285`

而常规 `use` 命令没有传 overwrite：

- `packages/cli/src/commands/openclaw/use.ts:49`

因此常规 `ccman openclaw use` 实际走的是 merge。

### 结论

这是一个**spec 与当前代码的冲突**：

- **代码**更符合你现在的需求（常规管理增量）
- **旧 spec**仍停留在“OpenClaw 切换即覆盖”的阶段

也就是说：

- 当前冲突不是“代码不符合你的目标”
- 而是“旧 spec 没有跟上新的统一写入策略”

## 6.2 历史脚本 `scripts/setup-gmn.mjs` 的代码与文档冲突

### 代码行为

- `scripts/setup-gmn.mjs:61`

```js
const provider = existing
  ? manager.edit(existing.id, { baseUrl, apiKey })
  : manager.add({ name: PROVIDER_NAME, baseUrl, apiKey })

manager.switch(provider.id, { mode: 'overwrite' })
```

也就是说，这个脚本最终是走 overwrite。

### 文档口径

- `scripts/README-GMN-SETUP.md:14`

文档写的是：

- `setup-gmn.mjs` 默认保护模式

甚至还把它列为“推荐使用”：

- `scripts/README-GMN-SETUP.md:241`
- `scripts/README-GMN-SETUP.md:252`

### 结论

这是一个非常直接的冲突：

- **代码**：快捷脚本最终 overwrite
- **文档**：默认保护模式

如果用户按文档理解，就会对实际写入行为产生错误预期。

## 6.4 `ccman gmn*` 的入口语义与内部实现存在轻微冲突

从入口语义上：

- `gmn` 是快捷配置
- 应该是 overwrite

从最终调用上看：

- `manager.switch(..., { mode: 'overwrite' })`

但从完整执行链路看：

- 先 `edit()`（可能触发 merge）
- 再 `switch(overwrite)`

### 结论

这不是文档冲突，
而是**实现细节冲突**：

- 表层语义是 overwrite
- 实际执行链路可能夹了一次 merge

这会让行为变得“不够纯”，也不够容易推理。

## 8. 建议的修正优先级

- 让快捷配置入口真正成为“纯 overwrite”
